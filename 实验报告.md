## 一、实验目的

本实验旨在通过构建一个完整的B2C电子商务平台，系统性地掌握Web全栈开发的核心技术流程。项目采用前后端分离架构，前端基于Vue 3框架与Element Plus组件库构建响应式用户界面，后端基于Spring Boot 3.3.6实现微服务化RESTful API设计。数据持久层采用MyBatis-Plus，通过LambdaQueryWrapper实现类型安全的链式查询，并配置PaginationInnerInterceptor分页插件实现高效数据分页。安全方面，系统基于Spring Security过滤器链与JWT（JSON Web Token）实现无状态身份认证，采用BCrypt加密算法保障用户密码安全，并通过RBAC（基于角色的访问控制）模型实现细粒度的动态权限管理。性能优化方面，利用Redis作为缓存中间件，对热点商品数据实施Cache-Aside缓存策略，对秒杀库存采用原子操作防止超卖。系统通过Docker Compose实现容器化一键部署，涵盖MySQL 8.0、Redis 7、以及四个微服务的完整编排。

通过本实验，重点培养以下能力：第一，掌握前后端分离架构下的数据交互模式、跨域解决方案与全局异常统一处理机制；第二，理解微服务架构中API网关路由转发、服务间通信、共享数据库与独立部署的设计思想；第三，熟练运用Redis缓存策略优化高并发场景下的系统性能；第四，具备将应用容器化部署、保障多环境一致性的工程实践能力。本项目的实现不仅是对课程所学Web开发技术的综合运用，更是一次接近企业级开发实践的深度训练，为后续从事Web全栈开发工作奠定了坚实基础。

## 二、实验内容

本实验实现了一个功能完备的电商平台，涵盖客户端购物全流程和管理后台数据管理两大核心场景，并在此基础上扩展了多个进阶功能模块，以全面体现Web全栈开发的技术深度。

### 2.1 客户端核心功能模块

**（1）用户认证与安全机制**

实现了基于JWT的无状态身份认证体系。用户登录时，后端使用HMAC-SHA256算法生成包含userId、username、role的JWT令牌，并支持"记住我"功能（7天有效期）。令牌存储于Redis（Key格式：`user:token:{userId}`），用于登出时主动失效。前端Vue Router的`beforeEach`导航守卫在用户访问受保护页面时进行Token校验，未登录用户自动跳转登录页，已登录用户访问登录/注册页自动跳转首页。Spring Security配置了无状态会话管理（`SessionCreationPolicy.STATELESS`），关闭CSRF防护，通过`JwtAuthenticationFilter`在每次请求中解析Token并注入SecurityContext。

> **注**：系统未选用传统的 Shiro 框架，而是采用了目前企业微服务架构下更主流、与 Spring Boot 生态契合度更高的 **Spring Security 6** 结合 JWT 实现无状态鉴权。Spring Security 6 对响应式编程（WebFlux）、OAuth 2.0 Resource Server 及方法级安全注解（`@PreAuthorize`）具有原生支持，更符合本项目微服务化、前后端分离的技术架构选型。

**（2）密码找回与邮件验证**

系统集成了Spring Mail邮件服务，通过网易163邮箱SMTP服务器（`smtp.163.com:465`）发送验证码。用户注册时需输入邮箱验证码（6位随机数字，Redis存储5分钟有效期），找回密码时同样通过邮箱验证码验证身份。后端`sendVerificationCode`方法使用`RandomUtil.randomNumbers(6)`生成验证码，通过`SimpleMailMessage`发送至用户邮箱，验证码校验通过后使用BCrypt加密新密码并更新数据库。

**（3）商品浏览与多维搜索 —— 对标淘宝的三栏式首页重构**

首页采用**对标淘宝的"三栏式"黄金版面布局**，突破了传统电商课程作业中常见的两栏设计局限。页面主容器宽度固化为**1320px**（即业界标准1200px的110%黄金放大比例），兼顾大屏饱满感与视觉呼吸感。在维持 PC 端 1320px 黄金版面的同时，系统底层引入了 CSS Media Queries（媒体查询）与 Element Plus 栅格系统（`<el-row :gutter="20">`）。当屏幕宽度小于 768px（移动端端点）时，三栏布局自动折叠为垂直流式布局（Flex Column），导航栏转换为汉堡菜单，各组件间距按比例收缩，完美实现多端响应式适配。三栏结构如下：

- **左侧（220px）**：商品分类树，支持多级分类展开，选择父分类时自动包含子分类商品。
- **中间（弹性宽度）**：焦点Banner轮播图（由管理员后台动态配置，支持排序），下方为热门商品（按销量降序排列）与新品推荐（按创建时间降序排列）卡片式网格。
- **右侧（275px）**：**用户/大促综合面板**（Right Panel），采用高度集成的白色卡片式设计，自上而下融合了"用户信息与核心资产"（头像/Hi问候语/资产快捷入口）、"平台快讯动态轮播列表"（限额3条，点击"更多"进入`/news`快讯详情页）、"领券中心与限时秒杀大促入口"三大功能模块。

商品列表页支持按分类筛选、关键词模糊搜索（同时匹配商品名称`name`和关键词`keywords`字段）。排序/筛选区摒弃了传统`<el-select>`下拉框组合，重构为**横向文本标签组（Tabs）**的无边界筛选体系——用户点击文本标签即可直切排序模式（默认 / 价格升序 ↓ / 价格降序 ↑ / 销量），配合`:hover`颜色过渡动画与小巧箭头图标，以极简的交互逻辑降低视觉负载，全面对标现代主流电商平台的筛选规范。商品详情页展示商品主图与多图轮播、规格选择、实时库存与销量信息，并集成了秒杀活动倒计时、可用优惠券展示等营销功能。

**业务流转与动效升维**：为了增强业务逻辑的闭环，首页"限时秒杀"等特定板块的"查看更多"按钮实现了**带参数的路由智能分发**。通过向`/products`路由传递`query: { type: 'flash_sale' }`参数，商品列表页能在初始化时精准拦截并过滤展示对应营销属性的商品。此外，所有核心商品卡片均引入了`transform: scale(1.1)`的平滑微交互动效（Micro-interactions），配合悬浮阴影，极大提升了用户的点击欲望与前台页面的生命力。

**全站品牌感UI注入**：系统引入了由蓝橙撞色、带有现代微笑曲线的"优选购"品牌Logo真图片，彻底淘汰纯文本标题形式。搜索组件重构为高亮纯白底色加细边框，右侧搜索按钮采用实心主色调胶囊按钮（pill形状无缝拼接），极大增强了输入与点击欲。搜索框下方动态加载基于 **Redis ZSet（有序集合）** 统计的"热门搜索词"标签，用户点击热词标签即可一键完成搜索，极大缩短了从浏览到检索的转化路径。全局背景色从纯白调整为高级灰（`#f4f4f4`），使纯白内容卡片通过微弱的`box-shadow`阴影"浮"于页面之上，层次分明，彻底消除了页面的扁平与廉价感。首页各业务板块（限时秒杀、新品推荐、促销活动等）的标题文本采用了**克制型渐变色（Accent Gradient）**渲染——通过将纯文本隔离至独立`<span>`节点，强制设定CSS渲染顺序（`background`优先声明），辅以`background-clip: text`与`!important`提权，解决了跨浏览器的实心色块渲染Bug，赋予标题极具大厂风格的色彩张力。右侧用户面板重构了身份标识的排版层级（Typography），利用字重（`font-weight`）与颜色对比提升VIP专属感，并通过`min-width: 0`、`overflow: hidden`与`text-overflow: ellipsis`组合防御机制，彻底解决了小屏/响应式状态下卡片文字被违规截断的布局Bug。CSS Flex布局深度调优（采用`justify-content: space-between`和`flex: 1; min-width: 0;`约束），完美解决了三栏高度对齐和按钮溢出截断问题，实现底部像素级完美平齐。

**（4）购物车与高级UI收银台**

基于Pinia全局状态管理实现购物车模块，包含`items`（购物车项列表）和`count`（商品总数）两个核心响应式状态，以及`fetchCart`、`addToCart`、`updateQuantity`、`removeItem`、`checkItem`、`checkAll`六个异步操作方法。购物车页支持全选/单选、批量删除、商品数量调整、实时小计计算。

**（5）订单全生命周期管理 —— C端卡片化UI重构**

订单状态流转路径为：`PENDING_PAY`（待支付）→ `PENDING_DELIVER`（待发货）→ `DELIVERED`（已发货）→ `COMPLETED`（已完成），异常分支包括`CANCELLED`（已取消）和`REFUNDING`（退款中）→ `REFUNDED`（已退款）。后端使用Hutool的雪花算法（`IdUtil.getSnowflakeNextIdStr()`）生成全局唯一订单号，通过`@Transactional`注解保证订单创建、购物车清空、库存扣减、优惠券核销、销量增加的原子性。

前端"我的订单"页面进行了**C端卡片化全面升维**：废弃了传统后台管理表格的展现形式，将订单列表重构为独立的白底大圆角**实体卡片布局（Card Layout）**。每张订单卡片顶部展示订单状态标签（通过`el-tag`组件按状态映射不同颜色），中部展示商品缩略图、名称与规格，底部通过Flex弹性布局对齐操作区高亮按钮（如"确认收货"、"去评价"、"取消订单"、"申请退款"），浅灰背景底色配合柔和`box-shadow`悬浮阴影营造层次感。多状态Tab切换面板（全部/待付款/待发货/已发货/已完成）基于Element Plus的`el-tabs`组件，每个Tab独立请求对应状态的订单列表。用户可在已发货/待收货状态下点击"申请退款"提交退款请求，后端处理退款审批后将订单状态更新为`REFUNDED`（已退款）。

订单详情页（Order Detail）引入了`<el-page-header>`组件构建全局返回导航，彻底消除了用户对浏览器强制后退按钮的依赖。页面核心区域使用`<el-steps>`步骤条组件实现订单状态的**动态可视化追踪**——当前已完成的步骤以主色调高亮，未完成的步骤以灰色占位，用户可一目了然地感知订单所处环节（已下单 → 已付款 → 已发货 → 已签收 → 已完成）。收货地址、商品明细、支付信息等模块拆分为无边界流式排版的独立白色卡片，每张卡片之间保留舒适的呼吸间距（`padding: 20px; margin-bottom: 12px`）。

**（8）数据库级数据注入（Data Seeding）与全栈状态机闭环**

### 2.3 进阶功能亮点（额外加分项）

**（1）全局异常统一处理机制**：系统通过`@RestControllerAdvice`注解的`GlobalExceptionHandler`类，对`BusinessException`（业务异常，返回自定义code和message）、`MethodArgumentNotValidException`/`BindException`（参数校验异常，返回400错误和字段级错误信息）、`AuthenticationException`（认证异常，返回401）、`AccessDeniedException`（权限异常，返回403）、`Exception`（未知异常，返回500并记录日志）五类异常进行全局统一拦截，确保前端始终收到结构一致的JSON响应（`Result<T>`格式），避免异常信息直接暴露给客户端。

**（2）统一响应格式设计**：系统定义了`Result<T>`泛型类（包含`code`、`message`、`data`三个字段）和`PageResult<T>`分页封装类（包含`total`、`page`、`size`、`totalPages`、`records`五个字段），所有API接口统一使用`Result.success(data)`返回成功响应，使用`Result.error(code, message)`返回错误响应，前端无需针对不同接口做差异化解析。

**（3）MyBatis-Plus自动填充与分页插件**：`MyMetaObjectHandler`实现了`insertFill`和`updateFill`方法，在插入和更新操作时自动填充`createTime`和`updateTime`字段，避免手动设置时间戳。`MybatisPlusConfig`配置了`PaginationInnerInterceptor`分页插件，所有`Page`对象查询自动进行物理分页，无需编写`LIMIT`子句。

**（4）雪花算法订单号生成**：使用Hutool的`IdUtil.getSnowflakeNextIdStr()`生成全局唯一、趋势递增的分布式订单号，避免了传统自增ID在分布式环境下的冲突问题，同时具备一定的安全防猜测性。

**（5）模拟物流轨迹追踪**：`LogisticsController`基于订单状态和物流单号，动态生成模拟物流轨迹节点（已揽收→运输中→到达中转站→派送中→已签收），已发货订单显示前3个节点，已签收订单显示全部5个节点，物流公司从顺丰、圆通、中通、韵达、EMS、京东物流中随机选择。

**（6）Knife4j API接口文档**：系统集成了Knife4j 4.5.0（基于Swagger/OpenAPI 3.0规范），通过`@Tag`和`@Operation`注解自动生成在线API文档，支持在线调试接口。访问`http://localhost:8080/doc.html`即可查看所有接口的请求参数、响应格式，并直接在页面上发起测试请求。

**（7）Gateway WebSocket透明代理**：`WebSocketProxyServer`在网关层实现了WebSocket的透明代理——客户端连接网关的`/ws/chat/{token}`端点，网关通过Jakarta WebSocket Client API与后端Order Service的`ChatWebSocket`建立连接，双向转发消息。网关不解析Token，由后端负责认证，实现了关注点分离。

**（8）CORS跨域解决方案**：`CorsConfig`配置了全局CORS过滤器，允许所有来源（`addAllowedOriginPattern("*")`）和所有HTTP方法，支持凭证传递（`setAllowCredentials(true)`），解决了前后端分离开发时的跨域问题。

**（9）静态资源智能路径解析**：`StaticResourceConfig`采用`walkUpFind`算法从应用启动目录向上遍历（最多5级）查找`frontend/dist`和`uploads`目录，解决了IDE直接运行、JAR包运行、Docker容器运行等不同部署场景下的路径兼容性问题。

**（10）ECharts数据可视化**：管理后台仪表盘使用ECharts 5.5 + vue-echarts 6.7渲染饼图（订单状态分布）和折线图（销售趋势），通过`ref`引用DOM容器，在`onMounted`中初始化图表实例，在`onBeforeUnmount`中销毁实例防止内存泄漏。

**（11）对标淘宝的三栏式首页重构**：彻底打破传统课程作业中常见的两栏布局，将首页重构为"**左侧商品分类树（220px） + 中间焦点轮播图（弹性） + 右侧用户/大促综合面板（275px）**"的标准三栏黄金版面。全站主容器宽度从保守的1200px升级为**1320px**（110%黄金放大比例），分类树、右侧面板、秒杀商品卡片同步按比例拓宽。通过CSS Flex布局深度调优（`justify-content: space-between`和`flex: 1; min-width: 0;`），完美解决了三栏"长短脚"高度不对齐和右下角按钮溢出截断的布局难题，实现底部像素级完美平齐。

**（12）大厂品牌感全站UI注入**：引入了由蓝橙撞色、带有现代微笑曲线的"优选购"品牌Logo真图片，彻底淘汰纯文本标题形式。搜索组件重构为高亮纯白底色加细边框，右侧搜索按钮采用实心主色调胶囊按钮（pill形状无缝拼接），增强用户输入与点击欲。全局背景色从纯白调整为高级灰（`#f4f4f4`），纯白内容卡片通过`box-shadow`阴影"浮"于页面之上，层次分明，彻底消除了页面的扁平与廉价感。这些设计细节体现了对现代电商UI设计语言（如淘宝、京东首页）的深度研究与工程化还原。

## 四、实验过程与分析

### 4.1 基于Pinia的购物车到收银台数据流转

**业务背景**：购物车是电商平台的核心交互模块，涉及商品选择、数量调整、金额计算、地址管理、优惠券抵扣到最终下单的完整数据链路。该链路跨越多个前端组件和后端API，必须保证数据一致性和用户体验的流畅性。

**技术实现**：

（1）**Pinia Store设计**：在`cart.js`中定义`useCartStore`，采用Composition API风格编写。Store包含`items`（购物车项列表，`ref([])`）和`count`（商品总数，`ref(0)`）两个核心响应式状态，以及6个异步操作方法。每个操作方法内部遵循"调用API → 重新拉取全量数据"的模式，确保前端状态与后端数据库保持最终一致性：

```javascript
const addToCart = async (data) => {
  await cartApi.add(data)   // 调用 POST /api/client/cart/add
  await fetchCart()          // 重新拉取 GET /api/client/cart/list
}
```

（2）**后端去重逻辑**：`CartServiceImpl.addToCart()`在添加商品时，先查询是否存在相同`userId + productId + spec`的购物车项：若存在则累加数量（`quantity + request.getQuantity()`），不存在则新建记录。该设计避免了购物车中出现重复商品行。

（3）**收银台状态管理**：`OrderConfirmDialog`组件使用`v-model:visible`双向绑定控制显示/隐藏，通过`props`接收`items`（商品列表）和`totalAmount`（总金额），通过`emit('submit', payload)`向父组件回传订单提交参数。组件内部维护了`addresses`（地址列表）、`coupons`（优惠券列表）、`selectedAddressId`、`payMethod`、`selectedCoupon`、`couponDiscount`等本地状态，优惠券选择后实时计算折扣金额。

（4）**数据流转全链路**：`Cart.vue` → 点击"去结算" → 弹出`OrderConfirmDialog`（传入已勾选商品） → 用户选择地址/支付方式/优惠券/填写备注 → 点击"提交订单" → `emit('submit')` → `Cart.vue`的`handleOrderSubmit`方法 → 调用`orderApi.create({addressId, payMethod, remark, couponId, couponDiscount})` → 后端`OrderServiceImpl.createOrder()`在`@Transactional`事务中完成校验、创建订单、扣减库存、清空购物车、核销优惠券 → 返回订单号 → 前端跳转订单详情页。

**分析**：该方案实现了单向数据流——用户操作触发Store方法，Store方法调用API，API返回后更新Store状态，UI自动响应式更新。Pinia的集中式状态管理避免了组件间通过Props/Events层层传递购物车数据，`OrderConfirmDialog`作为纯展示+交互组件，不直接修改全局状态，通过事件回传让父组件决定后续逻辑，体现了单一职责和关注点分离的设计原则。

### 4.2 Redis + MySQL双层防护的秒杀防超卖机制

**业务背景**：秒杀是电商平台最具挑战性的业务场景，核心难点在于高并发下如何保证库存扣减的原子性，防止超卖（即售出数量超过实际库存）。单纯依赖MySQL的行锁在高并发下性能瓶颈明显，需要引入Redis作为前置防护。

**技术实现**：

（1）**Redis库存预减**：在`SeckillServiceImpl.executeSeckill()`中，首先检查秒杀活动是否在有效期内，然后使用Redis的`opsForValue().decrement()`对Key格式为`seckill:stock:{promotionId}`的库存进行原子扣减。若Redis中不存在该Key，则从MySQL加载秒杀库存并写入Redis，同时设置TTL等于活动剩余时间。若扣除后库存小于0，立即执行`increment()`回滚并抛出"商品已售罄"异常。

```java
String stockKey = "seckill:stock:" + promotionId;
Boolean hasKey = redisTemplate.hasKey(stockKey);
if (Boolean.FALSE.equals(hasKey)) {
    int stock = promo.getSeckillStock() != null ? promo.getSeckillStock() : product.getStock();
    long ttl = Duration.between(now, promo.getEndTime()).getSeconds();
    if (ttl > 0) {
        redisTemplate.opsForValue().set(stockKey, String.valueOf(stock), ttl, TimeUnit.SECONDS);
    }
}
remaining = redisTemplate.opsForValue().decrement(stockKey);
if (remaining == null || remaining < 0) {
    redisTemplate.opsForValue().increment(stockKey);  // 立即回滚
    throw new BusinessException("商品已售罄");
}
```

（2）**数据库库存扣减**：Redis预减成功后，在`@Transactional`事务中执行数据库层面的库存扣减（`product.setStock(product.getStock() - quantity)`）和销量增加（`product.setSales(product.getSales() + quantity)`），通过`productMapper.updateById()`更新MySQL。若数据库操作失败导致事务回滚，catch块中执行`redisTemplate.opsForValue().increment(stockKey)`将Redis库存恢复，确保Redis与数据库数据一致。

（3）**容错处理**：当Redis不可用时，`decrement`操作被try-catch捕获，抛出"秒杀系统繁忙"提示，避免Redis故障导致整个秒杀流程崩溃。业务异常（`BusinessException`）直接抛出，不做Redis回滚。

**分析**：该方案采用"先Redis后MySQL"的扣减顺序，利用Redis单线程模型和原子操作的特性保证高并发下库存扣减的正确性，再通过数据库`@Transactional`事务保证数据持久化的一致性。Redis库存回滚机制（catch块中的`increment`）确保了异常情况下的数据补偿。这种双层防护策略在性能与正确性之间取得了良好平衡。

### 4.3 多表联查与Flex布局的真实头像评论区

**业务背景**：成熟的电商评论区不仅需要展示评价文字和星级，更需要展示用户头像和昵称，营造真实活跃的社区氛围。这要求后端在查询评价数据时关联用户表获取头像和昵称信息，前端则需重构UI以适配新的数据结构。

**技术实现**：

（1）**后端多表联查**：`ReviewServiceImpl.getProductReviews()`执行两步查询：
- 第一步：查询`review`表，条件为`product_id = ? AND status = 1`（仅已审核评价），按`create_time`降序排列。
- 第二步：遍历评价列表，对每条评价通过`userMapper.selectById(r.getUserId())`查询`user`表，获取`nickname`和`avatar`字段，组装到`ReviewVO`对象中。

```java
return reviews.stream().map(r -> {
    ReviewVO vo = ReviewVO.builder()
            .id(r.getId()).userId(r.getUserId())
            .content(r.getContent()).rating(r.getRating())
            .images(r.getImages()).createTime(r.getCreateTime())
            .build();
    User u = userMapper.selectById(r.getUserId());
    if (u != null) {
        vo.setUsername(u.getUsername());
        vo.setNickname(u.getNickname());
        vo.setAvatar(u.getAvatar());
    }
    return vo;
}).toList();
```

（2）**前端Flex布局渲染**：`ProductDetail.vue`的评论Tab采用经典左右布局：
- **左侧**（`flex-shrink:0; margin-right:12px`）：展示圆形头像（`width:40px; height:40px; border-radius:50%; object-fit:cover`），若`avatar`为空则使用`https://ui-avatars.com/api/?name=昵称`动态生成占位头像。
- **右侧**（`flex:1; min-width:0`）：第一行展示昵称（`font-weight:600; color:#303133`），第二行展示星级评分（`'★'.repeat(rating)`黄色渲染）与日期（`font-size:12px; color:#999`），第三行展示评价正文（`line-height:1.6; color:#333`），第四行展示评价图片缩略图（通过`parseImages`方法解析JSON数组）。
- 每条评价底部用`border-bottom:1px solid #f0f0f0`分隔，上下`padding:16px 0`。

（3）**发表评价的购买验证**：`ReviewServiceImpl.addReview()`在添加评价前，通过子查询验证用户是否已购买该商品（`SELECT COUNT(*) FROM order_item WHERE product_id = ? AND order_no IN (SELECT order_no FROM e_order WHERE user_id = ?)`），未购买用户无法发表评价，防止恶意刷评。

**分析**：该链路体现了前后端分离架构下典型的数据交互模式——前端声明式渲染、Gateway统一路由、后端Service层多表数据组装、DTO封装返回（ReviewVO包含User的nickname和avatar）。左右Flex布局兼顾了视觉美感和信息层次，蒸汽波UI风格（#f0f0f0分隔线、16px呼吸感padding）营造了专业的社区氛围。购买验证机制保障了评价的真实性。

在开发过程中，我遇到了若干典型的技术问题，以下选取两个最具代表性的Bug进行总结：

### Bug 1：静态资源 /uploads 路径映射导致图片无法加载（404错误）

**问题描述**：管理后台通过`/api/admin/site/upload`接口上传图片后，返回的图片URL为`/uploads/xxx.jpg`，但前端`<img>`标签加载该URL时返回404错误，图片无法显示。

**排查过程**：首先检查`uploads`目录下是否存在该文件（确认文件已成功写入），然后通过Postman直接请求`/uploads/xxx.jpg`（返回404）。分析发现Spring Boot默认只会映射Classpath下的`/static/`、`/public/`等路径，而`uploads`目录位于项目根目录下，不在Spring Boot默认的静态资源扫描范围内。

**解决方案**：在Gateway模块的`StaticResourceConfig`中实现`WebMvcConfigurer`接口，重写`addResourceHandlers`方法，通过`registry.addResourceHandler("/uploads/**").addResourceLocations(uploadsUrl)`将`/uploads/**`路径映射到文件系统中的`uploads`目录。同时采用`walkUpFind`算法从应用启动目录向上遍历查找`uploads`目录，增强了不同运行环境下的路径兼容性。修复后，上传的图片可正常在前端显示。

**知识点**：Spring Boot静态资源映射默认只扫描Classpath路径，文件系统中的目录需要手动配置`ResourceHandler`；`toFileUrl`方法返回的URL必须以`/`结尾，否则Spring的`UrlResource.createRelative()`会截断最后一段路径。

### Bug 2：全局缩放（Zoom）引发的组件库 DOM 传送门（Teleport）错位异常

**问题描述**：为提升大屏视觉张力，系统在根节点应用了`zoom: 1.1`全局缩放。但在该状态下，顶部导航栏的`<el-dropdown>`（用户菜单）和购物车悬浮提示框等弹出层组件，其定位出现了严重的横向与纵向偏移，甚至飞出可视区域。

**排查过程**：通过浏览器 DevTools 审查元素发现，Element Plus 为了防止弹出层被父元素的`overflow: hidden`遮挡，默认开启了`teleported`属性，将弹出层 DOM 节点直接挂载到了`<body>`的最外层。然而，全局`zoom`属性改变了整个文档的渲染坐标系，导致 UI 库内部依赖`getBoundingClientRect()`的绝对定位计算公式失效，缩放比例被错误叠加。

**解决方案**：针对受影响的悬浮/弹出组件（如`<el-dropdown>`、`<el-popover>`），显式关闭其传送门机制，添加`:teleported="false"`属性，强制这些弹出层 DOM 留在原父节点上下文中进行相对渲染。修复后，所有弹出层在 110% 甚至任意缩放比例下均能保持像素级精准定位。

**知识点**：深入理解了现代前端 UI 组件库的 Teleport（传送门）渲染机制，以及 CSS `zoom`与`transform: scale`在 DOM 坐标系计算中的底层差异与副作用。
